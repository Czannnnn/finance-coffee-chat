## 핵심 발견

1. **`의무보유확약 = 0.00%` 동형 패턴**
   시트 상위 5건 모두 동일하게 `0.00%` 저장. 일반 스팩은 실제 0.00%일 수 있으나, 일반 IPO 종목(매드업 등)까지 0.00%로 찍힌 것은 `ipo-scraper.gs:228-233`의 `lockupMatch` 정규식이 잘못된 섹션의 숫자를 잡고 있을 가능성이 높음.

2. **유통가능물량 / 경쟁률 / 확정공모가 광범위 공란**
   5건 중 이들 3필드가 한 건도 채워지지 않음. 실제 38.co.kr 상세 페이지가 비어있는 건인지, 파싱 실패로 빈 값이 저장된 건지 Apps Script 로그 없이 구분 불가 → 시트 저장 단계에서 이미 손실됨을 시사.

3. **희망공모가만 시트에 정확히 저장됨**
   목록 페이지의 `cells[3]` 텍스트를 그대로 가져오는 단순 로직이라 성공. 반면 상세 페이지 기반 필드는 모두 실패.

4. **38.co.kr 목록 1~3페이지에서 특정 종목 탐색 실패**
   레몬헬스케어·져스텍은 스팟체크 스크립트가 1~3p 범위 안에서 찾지 못함. 스크래퍼가 매일 `MAX_PAGES=2`만 수집하기 때문에 해당 종목들은 과거 수집분이 아직 시트에 남아있는 것. 종목 수명 사이클 관리 부재 — 상장 완료 종목 아카이브 없이 누적되기만 함.

5. **스팟체크 스크립트 자체 버그 (판단 무효화는 아님)**
   `find_detail_url`이 모든 종목에 대해 동일 URL(`no=2290` = 매드업)을 반환. row regex가 1차 매칭만 잡는 결함. 단 매드업 단독 결과만으로도 Apps Script 파싱 실패 패턴은 성립하므로 결론 불변.

## 결정 근거 확정

- **정규식 패치(P1-1/P1-2) 폐기 결정 정당**: 파싱 실패 범위가 3개 필드 × 대다수 행에 걸쳐 체계적이고, 38.co.kr 구조 변경이 재발할 위험도 상존.
- **P2 데이터 소스 재설계 채택 정당**: DART 전자공시(공식 JSON) 기반 산정이 `0.00%`/공란 문제를 근본 해소. 38.co.kr은 기관수요예측 경쟁률 보조 소스로만 유지.
- **P0 즉시 반영 가치**: 프론트 status 재계산(P0-3)·상장대기 신설(P0-4)은 데이터 소스와 무관한 버그 수정이므로 P2 이전에도 UX 개선 효과 있음.
