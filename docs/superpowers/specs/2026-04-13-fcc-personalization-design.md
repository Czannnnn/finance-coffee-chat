## 1. 배경 & 문제

사용자 피드백(2026-04-13):

1. **UX 라이팅** — `cost-optimization.html`의 "6 세그먼트 × 8 영역 매트릭스"라는 수학식 표현이 기계적이고 어려움
2. **개인화 부재** — 다른 앱에서도 지원금 정보를 확인하면 "내 조건(연봉·주택·부양가족)과 맞지 않는 내용이 대부분"인 경험이 반복됨. 필터링된 "딱 맞는" 정보를 보고 싶음

## 2. 목표 & 비목표

### 목표 (이번 스펙 범위)

- `cost-optimization.html`의 모든 "6×8 매트릭스" 용어를 친근한 자연어로 교체 (title / meta description / OG / Twitter Card / JSON-LD / 히어로 / 섹션 타이틀 포함)

- URL 쿼리 파라미터를 단일 상태 매체로 사용 — 서버·localStorage·sessionStorage·쿠키 **미사용**

- 세그먼트 × 영역 교차 전용 콘텐츠 (48조합 "빅 과제") — 별도 스펙

- A/B 테스트 인프라
- 추천 규칙 JSON의 전면 확장 — 이번 스펙에선 MVP 분량(영역당 2~3개 규칙, 총 16~24개 규칙)

## 3. 프라이버시 약속 (중요)

**기존:** `cost-simulation.html`은 "localStorage·sessionStorage·쿠키 어디에도 저장 안 함" 명시

**확장:**
- `cost-optimization.html` 허브도 **같은 수준**의 약속을 유지
- 필터 상태는 오직 URL 쿼리 파라미터에만 존재 → 브라우저 북마크/히스토리는 사용자 주권 영역이지 사이트의 저장 기능이 아님
- 필터 위젯 상단에 명시 고지:

  > 🔒 이 정보는 URL에만 담깁니다. 사이트나 브라우저 저장소에 저장되지 않으며, 탭을 닫는 순간 우리 쪽에는 어떤 흔적도 남지 않습니다.

- 공유 버튼 기본값: **민감정보 제외 ON** (연봉·부양가족 파라미터 제거한 URL 생성)
- `history.replaceState()` 사용 — 히스토리 오염 방지, back 버튼 1회로 이전 페이지 복귀 보장

## 4. URL 스키마

```
/cost-optimization.html
  ?s=s4            // 세그먼트 (s1~s6)            [필수]
  &income=50-70    // 연봉 구간                     [필수]
  &home=rent       // 주거                          [필수]
  &dep=1           // 부양가족                      [옵션]
  &region=metro    // 권역                          [옵션]
```

**필드 값 테이블:**

| 필드 | 허용 값 | 라벨 |
|---|---|---|
| `s` | `s1` | 청년 단독 |
| `s` | `s2` | 신혼 맞벌이 |
| `s` | `s3` | 신혼 외벌이 |
| `s` | `s4` | 유자녀 1자녀 |
| `s` | `s5` | 유자녀 다자녀 |
| `s` | `s6` | 중장년 단독 |
| `income` | `0-30` | 3천 이하 |
| `income` | `30-50` | 3천~5천 |
| `income` | `50-70` | 5천~7천 |
| `income` | `70-100` | 7천~1억 |
| `income` | `100+` | 1억 초과 |
| `home` | `own` | 자가 |
| `home` | `jeonse` | 전세 |
| `home` | `rent` | 월세 |
| `dep` | `0` / `1` / `2` / `3` | 부양가족 수 (3은 3명 이상) |
| `region` | `metro` | 수도권 |
| `region` | `nonmetro` | 비수도권 |

**파싱 규칙:**
- 빈 파라미터는 URL에서 생략
- 허용되지 않은 값은 무시하고 기본 허브 렌더 (에러 노출 없음)
- 필수 3필드 중 일부만 있으면 필터 위젯은 반영하되 맞춤 요약 섹션은 빈 상태

### 추천 규칙 JSON 구조

```json
{
  "version": "2026-04",
  "rules": [
    {
      "id": "rent-tax-credit",
      "title": "월세세액공제 신청",
      "savingHint": "연 최대 75만원 환급",
      "areaRef": "01-year-end-tax",
      "priority": 3,
      "conditions": {
        "home": ["rent"],
        "income": ["0-30", "30-50", "50-70"]
      },
      "linkHref": "cost/area-01-year-end-tax.html#월세공제",
      "linkLabel": "자세히 →"
    },
    {
      "id": "youth-rent-support",
      "title": "청년월세 지원금",
      "savingHint": "월 20만원 · 최대 12개월",
      "areaRef": "07-public-support",
      "priority": 3,
      "conditions": {
        "s": ["s1"],
        "home": ["rent"],
        "income": ["0-30", "30-50"]
      },
      "linkHref": "cost/area-07-public-support.html#청년월세",
      "linkLabel": "신청 경로 →"
    }
  ]
}
```

**스키마 설명:**
- `id`: 규칙 고유 식별자 (GA `click_recommendation` 이벤트 label로 사용)
- `title`: 카드 제목 (사용자 노출)
- `savingHint`: 예상 절감·혜택 금액 한 줄
- `areaRef`: 소속 영역 (카드 좌상단 뱃지 렌더용)
- `priority`: 1~3 정수 (3 = 최우선), 우선순위 점수 기초
- `conditions`: 필드명 → 허용 값 배열. 모든 필드 조건이 AND로 충족되면 매칭. 배열의 값 중 하나만 일치해도 OK
- `linkHref` / `linkLabel`: 상세 페이지 이동 링크

**매칭 및 정렬 알고리즘:**
1. 필터 상태의 필드만 추출 (옵션 필드가 없어도 동작)
2. 각 규칙에 대해 `conditions`의 모든 필드가 필터 상태와 매칭되는지 검사
3. 매칭되지 않은 규칙 중 "명확히 해당 없음"인 규칙(한 필드에서 조건이 정반대인 경우)은 "해당없음" 섹션으로 분리 수집
4. 매칭된 규칙을 점수순 정렬:
   - `score = priority × 10 + (옵션 필드 매칭 개수 × 3) + 세그먼트 가중치`
   - **세그먼트 가중치** = `cost-optimization.html` 우선순위 매트릭스 테이블에서 해당 규칙의 `areaRef`와 현재 `s` 세그먼트의 교차 셀 ★ 개수 (★★★ = 3, ★★ = 2, ★ = 1)
5. **상위 3개**는 TOP 3 대형 카드로 렌더, **4~8위**는 확장 리스트("💡 챙길 만한 것")로 렌더, **9위 이하**는 노출하지 않음
6. **해당없음 분류**: 어떤 규칙의 `conditions`에서 단일 필드만 정반대 매칭(예: `home: ["rent"]` 조건 vs 사용자 `home=own`)이라면 "해당없음" 섹션으로 분리. 복수 필드가 맞지 않으면 아예 제외


```
│ 가구 유형  [S1][S2][S3][S4][S5][S6]              │
│ 연봉 구간  [~3천][3~5천][5~7천][7~1억][1억+]    │
│ 주거      [자가][전세][월세]                     │
│                                                  │
│ ──── ▼ 더 정확히 보려면 2문항 더 ────           │
│ 부양가족  [0명][1명][2명][3명+]                  │
```

### 동작 규약

- **칩(chip) 방식** — 드롭다운보다 터치 친화적, 한 번에 전체 옵션 노출
- **즉시 반영** — 칩 클릭 즉시 `history.replaceState()`로 URL 갱신 + 재렌더. 제출 버튼 없음
- **토글 선택** — 같은 칩 다시 클릭 시 선택 해제
- **확장 가능 영역** — 옵션 2필드(부양가족·권역)는 디폴트 접힘, "더 정확히 보려면" 버튼으로 펼침
- **자동 펼침** — URL에 옵션 필드 값이 있으면 해당 영역 자동 펼침
- **초기화** — 모든 필터 제거, URL 파라미터 제거, 위젯 초기 상태
- **공유 버튼** — 현재 URL을 클립보드 복사. "민감정보 제외" 체크박스 기본 ON → `income`·`dep` 파라미터 제거한 URL 생성. 복사 성공 토스트 노출

### 반응형

- 데스크탑: 각 필드가 가로 한 줄, 칩 가로 나열
- 600px 이하: 각 필드 그룹 세로 스택, 칩 wrap, 터치 타겟 최소 44px 높이

### 해당없음 섹션의 의도

피드백의 "나한테 해당 안 되는 내용이 많았어서"를 **정반대**로 뒤집는 설계:
- 단순히 숨기지 않고, 명시적으로 "해당없음"으로 분리 표기
- 교육적 맥락 유지 ("이런 제도도 있는데 내 상황엔 해당 안 됨")
- 필터 조건을 바꾸면 해당없음 → 해당됨으로 이동하는 경험을 통해, 사용자가 규칙 구조를 직관적으로 이해하게 됨
- 신뢰감 형성: 추천이 "마법"이 아니라 "규칙 기반"이라는 것이 드러남

## 9. 기존 요소 재정렬

**필터 적용 시 기존 섹션 동작:**

- **세그먼트 카드 그리드 (기존 Section 1, 마크업 유지)**
  - 내 세그먼트(`s=s4`) 카드에 커피 테마 강조 테두리 + "내 유형" 배지 추가
  - 다른 카드는 그대로 유지 (회색/숨김 아님 — 탐색 가능성 유지)

- **영역 카드 그리드 (기존 Section 2)**
  - 내 세그먼트의 우선순위 매트릭스 행 참조 → ★★★, ★★, ★ 순으로 카드 재정렬
  - ★★★은 상단에 약간 크게, ★은 하단 회색 톤
  - 마크업은 JS로 순서만 재배열 (DOM 재삽입)

- **우선순위 매트릭스 테이블 (기존 Section 3)**
  - 내 세그먼트 행(`tr`)의 배경색을 커피 테마 하이라이트 (`--caramel` 알파 10%)
  - 테이블 자체는 그대로 유지 — 비교 자료로 가치 있음

**필터 미적용 시:** 기존 동작 그대로 유지, 재정렬 없음.

## 10. GA 이벤트 명세

**신규 이벤트 6개:**

| 이벤트명 | 트리거 | `click_text` | `click_location` |
|---|---|---|---|
| `click_filter_chip` | 필터 칩 클릭 | `{필드}:{값}` (예: `income:30-50`) | `cost_hub_filter` |
| `expand_filter_more` | "더 정확히 보려면" 펼침 | — | `cost_hub_filter` |
| `click_filter_reset` | 초기화 버튼 | — | `cost_hub_filter` |
| `click_filter_share` | 공유 버튼 | `masked` (민감정보 제외) 또는 `full` | `cost_hub_filter` |
| `click_recommendation` | 맞춤 요약 카드 클릭 | `{rule_id}` (예: `rent-tax-credit`) | `cost_hub_recommend` |
| `view_recommendation_rendered` | 요약 섹션 첫 렌더 | 필터 조합 시그니처 (예: `s4_50-70_rent`) | `cost_hub_recommend` |

**기존 이벤트 업데이트:**
- `click_segment` / `click_area`: 필터 적용 상태일 때 `data-ga-loc="cost_hub_filtered"`로 동적 주입. 기본 `cost_hub`는 미필터 상태 유지.

**구현 메모:**
- 기존 `index.html` 하단의 전역 delegation 스크립트(`document.addEventListener('click', ...)`)가 `cost-optimization.html`에도 동일하게 존재해야 함. `js/cost-personalizer.js`가 이 스크립트 로딩 이후에 실행되거나, 동일한 로직을 직접 포함
- `view_recommendation_rendered`는 `gtag('event', ...)`로 직접 발화 (클릭이 아닌 렌더 이벤트)
- `click_filter_chip`은 칩의 `data-ga-text`를 JS가 동적 갱신 (선택 값 반영)
